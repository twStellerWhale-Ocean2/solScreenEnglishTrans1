# I. 主旨目的

定義建置單元需要「取得指定線上影片（YouTube 等）之字幕（逐句文字＋起訖時間），供下游作為文字擷取來源」此一功能型態時的統一選型。**型態即契約身份、標準產品寫於內容**；日後換字幕來源只改本契約一處，引用之 design 不動。於 design.md ＜III 系統設計.C.(A) 技術選型＞引用並落地版本/用法。此型態專責**字幕文字取得**，與 [techItem影片播放] 之**播放控制**責任區隔——**僅取字幕文字、不下載影片內容**。

# II. 參考準備

* **統一標準產品（epic #178 增量6′-B「時間 pivot」定案）**：**字幕來源＝字幕檔本身**，取得端二選一——(a) **本機 `curl`（CLI）**：經 `System.Diagnostics.Process` 啟動 GET **字幕檔／逐字稿 URL** 之原文（Fandom/Cloudflare 等以 TLS 指紋擋 `HttpClient`〔即使帶完整瀏覽器標頭仍 403〕，curl 之指紋獲放行；curl.exe 隨 Windows 10 1803+ 內建、非另裝）；(b) **本機自製字幕檔**：唯讀讀取使用者所選檔（`OpenFileDialog`／拖放）。原文由宿主 `SubtitleParser` 純函式解析為逐句 cue（文字＋起始秒＋說話人）——**時間與說話人皆取自字幕檔自帶**（VTT／SRT 之 `-->` 時間行與 `<v 名字>`／`NAME:` 標記、或 fandom 式 `HH:MM:SS` 逐字稿）。**唯無時間戳之網頁式逐字稿**才落 AI 直接抽取（`ITranscriptAligner.ExtractTimedCuesAsync`：讀網頁純文字、逐句照抄原有時間戳，會花 OpenAI 額度、跑前確認）。
* **替代界線**：一律**僅取字幕文字、不下載影片本體、不抽音檔**（不繞過平台、不重散佈）。欲改用平台官方字幕 API 或其他取得端時於本契約換標準產品一處、design 引用不動（維持「字幕來源→逐句 cue」之取得介面邊界）。**已廢止之取得端（史）**：原「本機 `yt-dlp` 取 YouTube 字幕」與增量5′「字幕檔＋Whisper 聲音對齊」——前者受 YouTube 字幕限制、grey-screen 卡死；後者實測會把句序弄亂——皆於增量6′-B 廢止，改採「字幕檔自帶時間＋說話人」。
* **能力／成本前置**：`curl` 須可得（Win10 1803+／目標 ≥19041 內建；缺失時明確提示）；字幕檔須可取得（URL 可連線／本機檔可讀），**取得逾時／非零離開碼／檔案不存在或無權限一律明確錯誤並中止該片、不當機**；自帶時間戳者免費解析、零 API 花費，僅無時間戳之網頁式逐字稿落 AI 抽取（跑前確認費用）。字幕檔來源由使用者自負（比照自備金鑰）；本機字幕檔**唯讀存取、不修改/回寫/移動使用者原檔**。

# III. 內容程序

* **呼叫慣例**：curl 以 `Process` 非同步啟動、擷 stdout/stderr、設逾時；本機檔以非同步唯讀 IO；皆不阻塞 UI；失敗（非零離開碼、逾時、檔案不存在/無權限、取得不到）明確降級為可讀錯誤與下一步。
* **輸入／輸出**：輸入＝字幕檔／逐字稿 URL（走 curl）或本機字幕檔路徑＋語言偏好（英文）；輸出＝逐句 cue 清單（文字＋起始秒＋說話人），供播放到點暫停與點字查詢。
* **參數**：curl 路徑、逾時、字幕檔大小上限/批次上限走組態/常數；不硬編。
* **可驗證性**：字幕解析（VTT/SRT／fandom 式逐字稿→cue 之 `SubtitleParser`、AI 抽取回應解析之 `TranscriptAlign` 純函式）以固定字幕樣本／假 Responses JSON 單元測試；`curl` 呼叫與檔案 IO 以假 process／暫存檔注入攔截（不打真網路、不起真 CLI）。
* **隱私/合規**：**僅取字幕文字、不下載影片內容、不儲存影片**；來源由使用者自負。
